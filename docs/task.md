It is a constraint. Tension (auto-detected from Q answers) Impact Consequence Scale > Medium + Budget = Low -25 Critical Unsustainable cost structure. Force scope reduction or budget increase. Long jobs (Q-07=long) + Serverless deploy -25 Critical Timeout mismatch. Mandates background worker process. Multi-tenant (Q-02) + Single DB schema -20 High Data isolation risk. Mandates schema-per-tenant or RLS. Public API (Q-11) + No versioning plan -15 High Breaking changes become critical incidents. IF: stabilityScore >= 70 AND scale >= Medium AND team >= 4 USE: Modular Monolith with clear service boundaries Services are modules, not processes. Each module owns its DB tables, types, and service layer. No cross-module direct imports — only through defined interfaces. Can extract to microservices later with zero refactoring. IF: stabilityScore 50–69 OR team <= 3 OR timeline <= 2 weeks USE: Structured Monolith — single module with enforced layers One codebase. Strict folder layers. Import rules enforced by linting. Shared DB schema. Still production-grade — just no module isolation. This is the correct choice for most startups. IF: stabilityScore < 50 USE: BLOCKED — Reduce scope before proceeding Score below 50 indicates contradictory constraints. No architecture can succeed. Revisit Q-01 through Q-15 and resolve contradictions first. IF: stabilityScore >= 85 AND scale >= Large AND team >= 8 AND timeline >= 3 months USE: Microservices — only if ALL conditions met Each service has its own repo, DB, CI/CD, and team. Communication via typed contracts (gRPC or REST + OpenAPI). No shared DBs. No shared libraries without versioning. This is rarely correct for new projects.   02 DATABASE & ORM RULES Driven by Q-02, Q-05, Q-08. No personal preference. 2.1 Database Selection DECISION Which database engine? IF: Q-05 = Relational AND Q-02 = Multi-tenant AND Q-08 has compliance USE: PostgreSQL (Neon serverless or RDS) Row-Level Security for tenant isolation. Encrypted at rest. Point-in-time recovery. Connection pooling via PgBouncer or Neon built-in. Never use SQLite in production for multi-user systems. IF: Q-05 = Relational AND Q-02 = Single-tenant OR internal tool USE: PostgreSQL (Neon free tier or Supabase) Standard Postgres. No RLS needed. Schema migrations via ORM. Read replica only if Q-04 >= 10K users. IF: Q-05 = Document AND Q-06 = real-time AND Q-02 = single-tenant USE: MongoDB Atlas OR Firestore Use Firestore if deploying to Firebase/GCP and real-time sync is core. Use MongoDB Atlas otherwise. Never self-host MongoDB for a team < 5. IF: Q-05 = Time-series USE: TimescaleDB (Postgres extension) OR InfluxDB TimescaleDB preferred if the rest of the data is relational. Keeps one DB engine. InfluxDB only if time-series is the ONLY data shape. 2.2 ORM Selection DECISION Which ORM? IF: Database = PostgreSQL AND language = TypeScript USE: Drizzle ORM Schema-as-code. Type-safe query builder. Thin abstraction — SQL is visible. Migration files are plain SQL. No magic. Import only from db/index.ts. Never instantiate Drizzle client outside db/. IF: Database = PostgreSQL AND team prefers active-record AND scale < 100K USE: Prisma Richer DX but heavier runtime. Use Prisma Accelerate for connection pooling. Acceptable tradeoff for small teams. Same import rules apply. IF: Database = MongoDB USE: Mongoose OR Prisma with MongoDB adapter Mongoose for document-centric logic. Prisma for teams already using it elsewhere. Never use raw MongoDB driver in application code. 2.3 Multi-Tenancy Strategy DECISION How is tenant data isolated? IF: Q-02 = Multi-tenant AND compliance is present AND scale >= Medium USE: Schema-per-tenant (Postgres schemas) Each tenant gets their own Postgres schema. Full isolation. Migration must run per-tenant. Higher ops cost. Required for HIPAA/SOC2. IF: Q-02 = Multi-tenant AND no compliance AND scale <= Medium USE: Row-Level Security (RLS) on shared schema tenant_id column on every tenant-scoped table. RLS policy enforced at DB level. Application must set current_tenant_id on each connection. Never filter by tenant_id only in application code — DB must enforce. IF: Q-02 = Single-tenant USE: No isolation needed — single schema Still add created_by and owner_id on all resource tables for ownership checks. 2.4 Database Hard Rules RDB-01 [HARD] DB client is instantiated ONCE in db/index.ts. Nowhere else. Any file outside db/ that imports the DB client directly is a linting violation. Repositories are the only callers of db/. RDB-02 [HARD] Every table has: id (UUID), created_at, updated_at, is_deleted (if soft-delete applies). No table ships without these four columns. Migrations that omit them are rejected. RDB-03 [HARD] Foreign keys are enforced at the DB level — not just in application code. No FK without a DB-level constraint. Drizzle references() or Prisma @relation both generate the constraint. RDB-04 [HARD] Raw SQL is forbidden in service and domain layers. Only repositories may write queries. If raw SQL is needed, it lives in a repository method with a comment explaining why. RDB-05 [HARD] Migrations are auto-generated by the ORM and never hand-edited. If a migration file is edited manually, it is invalid. Drop and regenerate. Hand-edits introduce schema drift. RDB-06 [SOFT] No JSONB column without a documented schema and Zod validator. JSONB is acceptable for agent outputs, config blobs, and flexible metadata. It is never acceptable for first-class domain data that will be queried.   03 ACCESS CONTROL MODEL RBAC, ABAC, or Policy-based — chosen from Q-03, not assumed 3.1 Model Selection DECISION Which access control model? IF: Q-03 = Simple roles AND Q-02 = Single-tenant AND roles <= 4 USE: RBAC — Role-Based Access Control User has one role. Role grants a set of permissions. Roles are static enums. Permission check: hasRole(user, Role.ADMIN). Enforcement in middleware and service layer. Never in UI only. IF: Q-03 = Attribute-based OR Q-02 = Multi-tenant OR resource ownership is complex USE: ABAC — Attribute-Based Access Control Decisions based on: subject attributes (user.role, user.orgId), resource attributes (resource.ownerId, resource.status), environment (time, IP). Policy: can(user, 'read', resource). Use CASL.js for implementation. Define abilities in lib/auth/abilities.ts. IF: Q-03 = Policy-based OR Q-11 = public API with scopes OR Q-08 = SOC2/HIPAA USE: Policy-Based (OPA or custom policy engine) Policies are declarative, versioned, and testable. Each policy is a pure function: (subject, action, resource, context) => boolean. Policies live in lib/auth/policies/. Every policy has unit tests. Audit log records every policy evaluation. IF: Q-03 = Ownership-only AND Q-02 = Single-tenant USE: Ownership model — no formal RBAC Resource has owner_id. User can act on resource if user.id === resource.ownerId. Admin override via is_admin flag. Simple, transparent, zero framework overhead. 3.2 RBAC Rules (if RBAC chosen) Rule Detail Role definition Roles are TypeScript enums. Defined once in types/auth.types.ts. Never as raw strings. Permission matrix Explicit matrix: Role → Action → Resource. Defined in lib/auth/permissions.ts. No implicit grants. Enforcement layers 1) Next.js middleware (route protection) 2) Service layer (business logic) 3) Repository layer (data filter). Never UI-only. No role escalation A user cannot grant a role higher than their own. Validated server-side on every role assignment. Admin override Admin role bypasses ownership checks, not security checks. Admin still cannot write to read-only resources. Role persistence Role stored in users table. Never in JWT only — JWT can become stale. Verify against DB on sensitive operations. 3.3 ABAC Rules (if ABAC chosen — CASL.js) // lib/auth/abilities.ts — ability definition pattern import { AbilityBuilder, createMongoAbility } from '@casl/ability'; import type { User } from '@/types/auth.types'; export function defineAbilitiesFor(user: User) { const { can, cannot, build } = new AbilityBuilder(createMongoAbility); if (user.role === 'admin') { can('manage', 'all'); // admin: unrestricted } else if (user.role === 'member') { can('read', 'Project', { orgId: user.orgId }); // own org only can('create', 'Project', { orgId: user.orgId }); can('update', 'Project', { ownerId: user.id }); // own resources cannot('delete', 'Project'); // explicit deny } return build(); } // Usage in service: const ability = defineAbilitiesFor(session.user); if (ability.cannot('delete', project)) throw new ForbiddenError(); 3.4 Auth System Hard Rules RAUTH-01 [HARD] Auth check is the FIRST operation in every Route Handler after input validation. No business logic executes before session is verified. Not first-ish. First. RAUTH-02 [HARD] Access control is enforced in the service layer, not only in route handlers. Route handlers can be bypassed. Service layer cannot. Ownership and permission checks live in services. RAUTH-03 [HARD] Tokens are never stored in localStorage. Use httpOnly cookies. localStorage is accessible to XSS. httpOnly cookies are not. RAUTH-04 [HARD] JWT payload contains only: userId, role, sessionId, exp. No sensitive data. JWT is signed, not encrypted. Treat payload as public. Verify sensitive attributes against DB. RAUTH-05 [HARD] Session expiry is enforced server-side. Client-side expiry is UX only. A client can forge an expired token check. Server always validates exp claim.   04 FOLDER STRUCTURE & IMPORT BOUNDARIES Adaptive tree — modules added per Q-answers IMPORT LAW — Memorise this. app/ → components/ → hooks/ → lib/domain/ → lib/services/ → lib/repositories/ → db/ agents/ → lib/services/ → lib/repositories/ → db/ Any import that goes AGAINST this flow is a linting error and a blocked PR.
